# 2026-04-03 会员余额消费记录功能

## 修改日期
2026-04-03

## 修改内容
在结账扣除会员余额后，自动插入消费记录到 `biz_consumption_record` 表，确保会员余额变动有完整的流水记录。

## 问题描述
之前在结账扣除会员余额时，只更新了会员表的余额字段，但没有在消费记录表中插入流水记录，导致：
1. 无法追溯会员余额的历史消费明细
2. 财务对账时缺少凭证
3. 会员查询余额变动记录时数据不完整

## 功能说明

### 业务流程
1. **结账时**：系统判断支付方式
2. **余额支付**：如果选择了"会员余额"或"组合支付"
3. **扣除余额**：从会员账户扣除相应金额
4. **插入记录**：在消费记录表中插入流水
5. **记录详情**：
   - 会员基本信息（ID、姓名、手机号）
   - 关联订单ID
   - 消费金额
   - 消费前后余额
   - 备注信息（包含订单号）
   - 创建时间（自动填充）

### 支付方式说明
- **balance（会员余额）**：全额从余额扣除，消费金额 = 订单金额
- **combined（组合支付）**：余额 + 线下组合，消费金额 = 实际扣除的余额金额

### 消费记录特点
1. **完整性**：记录了消费前后的余额，便于对账
2. **可追溯**：关联订单ID，可追溯消费来源
3. **自动化**：扣除余额时自动插入，无需手动操作
4. **事务保护**：在 `@Transactional` 事务中执行，确保数据一致性

## 优势
1. ✅ **数据完整性**：余额变动有完整的流水记录
2. ✅ **财务对账**：提供准确的消费凭证
3. ✅ **会员查询**：会员可查看余额变动明细
4. ✅ **审计追溯**：每笔消费都有据可查
5. ✅ **事务一致性**：在事务中执行，确保数据准确性

## 测试要点
1. **余额支付**：验证消费记录金额等于订单金额
2. **组合支付**：验证消费记录金额等于实际扣除的余额
3. **余额充足**：验证 `balance_after = balance_before - amount`
4. **订单关联**：验证 `order_id` 正确关联
5. **并发场景**：验证事务隔离性，避免数据不一致
