# 2026-04-03 会员余额不足验证功能

## 修改日期
2026-04-03

## 修改内容
在开始计时和续费时，如果选择了会员并使用"会员余额"支付方式，系统会验证会员余额是否足够支付订单金额。如果余额不足，会弹出提示框并阻止操作。

## 问题描述
之前的系统没有在开始计时时验证会员余额是否足够。当会员余额不足时，用户可以选择"会员余额"支付方式并开始计时，但结账时才发现余额不足，导致：
1. 用户体验差：开始时不知道余额不足，结账时才发现
2. 业务逻辑不合理：余额不足不应该允许选择该支付方式
3. 容易产生纠纷：用户以为可以支付，结果结账时出现问题

// 验证会员余额是否足够（仅对会员余额支付方式）
if (selectedMember.value && selectedPaymentMethod.value === 'balance') {
  const memberBalance = selectedMember.value.balance || 0
  const finalAmount = discountedPrice.value || 0

## 功能说明

### 验证时机
1. **开始计时时**：用户点击"开始计时"按钮时
2. **续费时**：用户点击"确认续费"按钮时

### 验证条件
- 已选择会员
- 支付方式为"会员余额"（`balance`）
- 会员余额 < 订单金额

### 支付方式说明
- **线下支付 (offline)**：不需要验证余额
- **线上支付 (online)**：不需要验证余额
- **会员余额 (balance)**：**需要验证余额是否足够**
- **组合支付 (combined)**：不需要验证（余额不足时自动拆分）

### 提示信息格式

**开始计时场景**：
```
会员余额不足！
当前余额：¥50.00
需要支付：¥68.00
还差：¥18.00
```

**续费场景**：
```
会员余额不足！
当前余额：¥30.00
总需支付：¥54.00（当前订单¥35.00 + 续费¥19.00）
还差：¥24.00
```

## 优势
1. ✅ **用户体验提升**：在操作前就知道余额不足，避免结账时才发现
2. ✅ **业务逻辑合理**：余额不足时不允许选择该支付方式
3. ✅ **减少纠纷**：清晰展示金额明细，用户明明白白消费
4. ✅ **操作友好**：弹窗提示，点击"知道了"即可关闭，可切换其他支付方式
5. ✅ **信息详细**：展示当前余额、需要金额、还差金额，一目了然

## 用户操作流程

### 场景 1：开始计时 - 余额不足
1. 用户选择会员（余额 50 元）
2. 选择套餐（68 元）
3. 选择支付方式为"会员余额"
4. 点击"开始计时"
5. 系统弹窗提示"余额不足"
6. 用户点击"知道了"关闭弹窗
7. 用户可以切换支付方式为"组合支付"或"线下支付"

### 场景 2：续费 - 余额不足
1. 当前订单 35 元，会员余额 30 元
2. 用户选择续费套餐 19 元，总金额 54 元
3. 选择支付方式为"会员余额"
4. 点击"确认续费"
5. 系统弹窗提示"余额不足"，展示明细
6. 用户点击"知道了"关闭弹窗
7. 用户可以切换支付方式为"组合支付"或"线下支付"

### 场景 3：余额充足
1. 用户选择会员（余额 100 元）
2. 选择套餐（68 元）
3. 选择支付方式为"会员余额"
4. 点击"开始计时"
5. 验证通过，直接开始计时
6. 结账时扣除 68 元，剩余 32 元

## 测试要点
1. **开始计时 - 余额不足**：验证弹窗提示正确，阻止开始
2. **开始计时 - 余额充足**：验证正常开始计时
3. **续费 - 余额不足**：验证弹窗提示正确，展示明细，阻止续费
4. **续费 - 余额充足**：验证正常续费
5. **其他支付方式**：验证不会触发余额验证
6. **无会员**：验证不会触发余额验证
7. **弹窗关闭后操作**：验证可以切换支付方式并继续操作
