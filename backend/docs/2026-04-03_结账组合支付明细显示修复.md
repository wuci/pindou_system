# 2026-04-03 结账组合支付明细显示修复

## 修改日期
2026-04-03

## 问题描述
在结账界面选择组合支付时，没有看到支付明细的展示效果。经过排查发现问题原因：

**根本原因**：
- 在订单进行中（还未结账）时，订单中的 `balanceAmount` 和 `otherPaymentAmount` 字段都被设置为 0
- 只有在结账时（`endTimer` 方法），才会根据会员余额和订单金额计算并更新这两个字段
- 获取账单时（`getBill` 方法），后端直接返回订单中的值，导致前端收到的是 0
- 前端条件判断 `bill.balanceAmount > 0 || bill.otherPaymentAmount > 0` 不成立，所以不显示支付明细

## 修复方案

在获取账单时（`getBill` 方法），如果支付方式是 'combined' 或 'balance'，并且订单中还没有设置实际支付金额（为 0 或 null），则**动态计算**并返回预计的支付拆分。

修改后：
```java
// 设置支付方式信息
if (order.getPaymentMethod() != null) {
    response.setPaymentMethod(order.getPaymentMethod());

    // 对于组合支付或余额支付，如果订单中还没有设置支付金额，则动态计算
    if (("combined".equals(order.getPaymentMethod()) || "balance".equals(order.getPaymentMethod())) &&
        (order.getBalanceAmount() == null || order.getBalanceAmount().doubleValue() == 0) &&
        order.getMemberId() != null) {

        try {
            // 查询会员余额
            Member member = memberMapper.selectById(order.getMemberId());
            if (member != null && member.getBalance() != null) {
                double totalAmount = amountDetail.getTotalAmount();
                double balance = member.getBalance().doubleValue();

                // 计算支付拆分
                double balanceAmount;
                double otherAmount;

                if ("balance".equals(order.getPaymentMethod())) {
                    // 余额支付：全额使用余额
                    balanceAmount = totalAmount;
                    otherAmount = 0;
                } else {
                    // 组合支付
                    if (balance >= totalAmount) {
                        // 余额充足，全部用余额
                        balanceAmount = totalAmount;
                        otherAmount = 0;
                    } else {
                        // 余额不足，用完余额，剩余线下支付
                        balanceAmount = balance;
                        otherAmount = totalAmount - balance;
                    }
                }

                response.setBalanceAmount(balanceAmount);
                response.setOtherPaymentAmount(otherAmount);

                log.info("支付方式动态计算 - 支付方式:{}, 订单总额:{}, 会员余额:{}, 余额支付:{}, 线下支付:{}",
                        order.getPaymentMethod(), totalAmount, balance, balanceAmount, otherAmount);
            }
        } catch (Exception e) {
            log.warn("动态计算支付金额失败: {}", e.getMessage());
            // 计算失败，使用订单中的值
            response.setBalanceAmount(order.getBalanceAmount() != null ?
                order.getBalanceAmount().doubleValue() : null);
            response.setOtherPaymentAmount(order.getOtherPaymentAmount() != null ?
                order.getOtherPaymentAmount().doubleValue() : null);
        }
    } else {
        // 非余额/组合支付或订单中已有支付金额，直接使用订单中的值
        response.setBalanceAmount(order.getBalanceAmount() != null ?
            order.getBalanceAmount().doubleValue() : null);
        response.setOtherPaymentAmount(order.getOtherPaymentAmount() != null ?
            order.getOtherPaymentAmount().doubleValue() : null);
    }
}
```

## 功能说明

### 动态计算逻辑

**触发条件**：
1. 支付方式是 'combined'（组合支付）或 'balance'（余额支付）
2. 订单中的 `balanceAmount` 为 null 或 0
3. 订单有会员ID

**计算规则**：

**余额支付（balance）**：
- `balanceAmount` = 订单总额
- `otherPaymentAmount` = 0

**组合支付（combined）**：
- 如果会员余额 >= 订单总额：
  - `balanceAmount` = 订单总额
  - `otherPaymentAmount` = 0
- 如果会员余额 < 订单总额：
  - `balanceAmount` = 会员余额
  - `otherPaymentAmount` = 订单总额 - 会员余额

### 显示效果

**场景 1：组合支付 - 余额充足**
- 订单总额：54 元
- 会员余额：100 元
- 显示：
  ```
  支付明细
  💰 会员余额支付  -¥54.00
  ```

**场景 2：组合支付 - 余额不足**
- 订单总额：54 元
- 会员余额：30 元
- 显示：
  ```
  支付明细
  💰 会员余额支付  -¥30.00
  🛒 线下支付      -¥24.00
  ```

**场景 3：余额支付**
- 订单总额：35 元
- 会员余额：100 元
- 显示：
  ```
  💰 会员余额支付  -¥35.00
  ```

## 数据流程

### 订单进行中（未结账）

1. **开始计时时**：
   - 订单创建
   - `paymentMethod` = 'combined'
   - `balanceAmount` = 0
   - `otherPaymentAmount` = 0

2. **获取账单时**：
   - 后端检测到支付方式是 'combined' 且金额为 0
   - 动态查询会员余额
   - 计算支付拆分
   - 返回计算后的金额
   - 前端显示支付明细

### 结账时

1. **结账请求**：
   - 后端实际扣除余额
   - 更新订单中的 `balanceAmount` 和 `otherPaymentAmount`
   - 保存实际支付金额

2. **再次获取账单**：
   - 后端直接返回订单中的实际金额（不再重新计算）

## 优势

1. ✅ **提前展示**：用户在结账前就能看到支付明细，提前知道如何扣费
2. ✅ **准确预估**：动态计算基于当前会员余额和订单金额，预估准确
3. ✅ **智能判断**：只在需要时才动态计算，不影响其他支付方式
4. ✅ **异常处理**：计算失败时使用订单中的值，确保系统稳定
5. ✅ **日志完善**：记录计算过程，便于排查问题
