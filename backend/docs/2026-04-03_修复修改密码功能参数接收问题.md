# 修复修改密码功能参数接收问题

**日期：** 2026-04-03
**修改人：** wuci
**问题描述：** 修改密码功能不起效，密码没有被正确修改

## 问题排查

### 问题现象
- 用户在前端点击"修改密码"
- 输入新密码并提交
- 提示"修改密码成功"
- 但使用新密码无法登录，原密码仍然有效

### 问题原因

**后端接口参数接收方式与前端请求方式不匹配**

#### 前端发送方式（`frontend/src/api/user.ts`）
```typescript
export const resetPassword = (id: string, password: string) => {
  return http.put(`/users/${id}/reset-password`, { password })
}

export const changePassword = (data: ChangePasswordParams) => {
  return http.put(`/users/${data.userId}/reset-password`, { password: data.newPassword })
}
```

前端发送的是 **JSON 请求体**：
```json
{
  "password": "新密码"
}
```

#### 后端接收方式（修改前）
```java
@PutMapping("/{id}/reset-password")
public Result<Void> resetPassword(
        @PathVariable("id") String userId,
        @RequestParam(defaultValue = "123456") String newPassword) {

    userService.resetPassword(userId, newPassword);
    return Result.success();
}
```

后端使用 `@RequestParam` 接收参数，期望的是 **URL 查询参数**：
```
PUT /users/{id}/reset-password?newPassword=新密码
```

### 参数不匹配分析

| 方面 | 前端 | 后端（修改前） | 匹配情况 |
|-----|------|--------------|---------|
| 参数位置 | 请求体 Body | URL 查询参数 | ❌ 不匹配 |
| Content-Type | application/json | application/x-www-form-urlencoded | ❌ 不匹配 |
| 参数名 | `password` | `newPassword` | ❌ 不匹配 |
| 默认值 | 无 | "123456" | - |

**结果**：后端一直使用默认值 "123456"，前端传递的新密码被忽略了。

## 解决方案

### 1. 创建重置密码请求DTO

**文件：** `backend/pdsystem-entity/src/main/java/com/pindou/timer/dto/ResetPasswordRequest.java`

```java
package com.pindou.timer.dto;

import io.swagger.v3.oas.annotations.media.Schema;
import lombok.Data;

import javax.validation.constraints.NotBlank;
import javax.validation.constraints.Size;
import java.io.Serializable;

/**
 * 重置密码请求DTO
 *
 * @author wuci
 * @date 2026-04-03
 */
@Data
@Schema(description = "重置密码请求")
public class ResetPasswordRequest implements Serializable {

    private static final long serialVersionUID = 1L;

    @Schema(description = "新密码")
    @NotBlank(message = "密码不能为空")
    @Size(min = 6, max = 20, message = "密码长度在 6 到 20 个字符")
    private String password;
}
```

### 2. 修改后端控制器接口

**文件：** `backend/pdsystem-server/src/main/java/com/pindou/timer/controller/UserController.java`

```java
/**
 * 重置用户密码
 */
@Operation(summary = "重置用户密码")
@PutMapping("/{id}/reset-password")
public Result<Void> resetPassword(
        @Parameter(description = "用户ID") @PathVariable("id") String userId,
        @Validated @RequestBody ResetPasswordRequest request) {

    log.info("重置用户密码请求: userId={}", userId);

    userService.resetPassword(userId, request.getPassword());

    return Result.success();
}
```

### 修改要点

1. **参数接收方式**：从 `@RequestParam` 改为 `@RequestBody`
2. **参数类型**：从 `String` 改为 `ResetPasswordRequest` 对象
3. **参数获取**：从直接使用 `newPassword` 改为 `request.getPassword()`
4. **参数验证**：添加 `@Validated` 注解，启用 DTO 中的验证规则
5. **移除默认值**：不再使用默认值 "123456"，由前端传递

## 技术要点

### @RequestBody vs @RequestParam

| 注解 | 参数位置 | Content-Type | 使用场景 |
|-----|---------|-------------|---------|
| @RequestParam | URL 参数 | application/x-www-form-urlencoded | 简单参数 |
| @RequestBody | 请求体 | application/json | 复杂对象 |

### 为什么使用 @RequestBody

1. **安全性**：密码放在 URL 中不安全，会记录在访问日志中
2. **规范性**：修改操作应该使用请求体，符合 RESTful 规范
3. **扩展性**：如果未来需要添加额外字段（如旧密码验证），无需修改接口签名
4. **验证**：可以使用 JSR-303 验证注解进行参数验证
